# Architecture Decisions

Short records of the key decisions behind the runtime. Each one is the "why" for a pattern you'll see throughout the code.

## ADR-001: Vercel AI SDK over direct provider SDKs

**Decision:** Use the Vercel AI SDK (`ai`, `@ai-sdk/anthropic`, `@ai-sdk/openai`, `@ai-sdk/google`) for all LLM calls instead of maintaining our own provider adapters.

**Context:** Every LLM provider has slightly different streaming formats, token-counting shapes, tool-call conventions, and error types. Previously we maintained ~1000 lines of adapter code translating between a unified shape and each provider's quirks. Every provider update was a code change.

**Rationale:** The AI SDK handles all provider-specific quirks natively — Anthropic prompt caching, tool call interleaving, token usage with cache breakdowns, streaming formats — without us writing adapter code. It's ~50 KB, TypeScript-first, actively maintained, and adding a new provider is a one-line change. The tradeoff is pinning to their release cycle; we accept it because the adapter layer is thin enough to swap out if needed.

## ADR-002: State machine over imperative loop

**Decision:** Implement the agent loop as an explicit discriminated-union state machine rather than a `while` loop with implicit states hidden in flags and branches.

**Context:** Most agent frameworks use a simple `while (true) { response = await call(); if (!toolCalls) break; runTools(); }` loop. It's easy to read but breaks down when you need to pause mid-loop for user confirmation, insert compaction or loop detection, spawn sub-agents, or add new states without tangling the control flow.

**Rationale:** Every state is explicit. Each handler is a pure `(state, ctx) → { next, effects }` function that can be tested in isolation. Adding a new state is additive — existing handlers don't change. Exhaustiveness is enforced by the TypeScript `never` trick, so forgetting a case is a compile error, not silent fallthrough. The cost is more code upfront and a pattern some developers aren't familiar with. We think it's worth it because we hit every one of those "breaks down when" cases. See [State Machine](/learn/architecture/state-machine).

## ADR-003: `Result<T, E>` over throw/null for fallible operations

**Decision:** Functions that can fail return `Result<T, E>` instead of throwing exceptions or returning `null`/`undefined`.

**Context:** A codebase audit during the rewrite found 7 critical "silent swallow" `try/catch` blocks — code that returned `null` on database failures, so callers thought "no data" when the database was actually broken. This pattern caused multiple multi-hour debugging sessions with no observable error.

**Rationale:** `Result<T, E>` forces the caller to handle both branches. The type system makes it impossible to treat "not found" and "broken" the same way. Exceptions are reserved for truly unexpected conditions that should bubble to a module boundary; `Result` is for expected failures (not found, validation failed, unauthorized). Error boundaries exist only at module edges (API routes, tool executors, session manager) where errors become structured responses the agent or caller can reason about.

## ADR-004: `@amodalai/types` zero-dep package

**Decision:** Extract shared types into a zero-dependency `@amodalai/types` package that every other package can safely import.

**Context:** Our cloud platform's `platform-api` couldn't import `@amodalai/core` because core's barrel exports transitively pulled in tree-sitter WASM (via the page builder's AST parsing). This meant `platform-api` had to inline types and treat shared shapes as opaque `Record<string, unknown>` — losing all type safety at the platform/runtime boundary.

**Rationale:** Types flow downward. `types` is a leaf node with no runtime dependencies. Cloud services, ISV code, custom tool handlers — anything — can import from `@amodalai/types` without dragging in WASM, Node-only modules, or circular dependencies. Matches the pattern Vercel AI SDK uses (`@ai-sdk/provider` is their zero-dep types package).

## ADR-005: Drizzle ORM over raw SQL

**Decision:** Use Drizzle ORM for all store-backend SQL, across both PGLite (local) and Postgres (production).

**Context:** The PGLite backend originally used raw SQL string interpolation. During the security audit we found SQL injection via store field names — user-provided `filter` and `sort` field names were interpolated directly into the query. The `values` were parameterized but the *field names* weren't, and field names flow from LLM tool call arguments.

**Rationale:** Drizzle's schema-bound column references eliminate SQL string interpolation by construction. The same schema drives both PGLite and Postgres backends — no "write it once for sqlite, rewrite for Postgres." Drizzle's async client integrates naturally with the write queue (for PGLite's serialization requirement). The tradeoff is another dependency and a learning curve for contributors not familiar with Drizzle; we accept it because the alternative is a live vulnerability class.

## ADR-006: Integration tests over unit tests for tool execution

**Decision:** Tool execution is covered by integration tests (real store backend, real tool registry, real session) rather than unit tests with mocked dependencies.

**Context:** We had a silent-failure bug where custom tools calling `ctx.request()` returned a green checkmark in the UI for weeks while actually failing with "no base_url" — because unit tests mocked the connection lookup and didn't exercise the real field-name mismatch (`base_url` vs `baseUrl`).

**Rationale:** Mocks let bugs hide in the integration seams. An integration test that exercises the real path (tool → registry → executor → backend) catches shape mismatches, missing fields, and wiring errors that mocked unit tests never see. We reserve unit tests for pure utility functions. Tool execution, session management, and HTTP routes get integration coverage.

## ADR-007: Sub-agent dispatch over deep reasoning

**Decision:** Handle complex multi-step tasks by dispatching sub-agents with fresh context via the `dispatch_task` tool, rather than extending the primary agent's reasoning depth.

**Context:** Long agent runs bloat the primary's context with intermediate tool results, raising cost per turn and degrading judgment. The alternative is compacting aggressively, which loses detail.

**Rationale:** Sub-agents run in isolated state machines, make their own tool calls, and return a compressed summary to the parent. The parent's context stays clean — it sees "dispatched task, got summary" instead of 15 tool results. Sub-agents can write (unlike read-only "explore" agents) and have full tool access, so they're production-grade for data-gathering workflows.
