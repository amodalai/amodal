# Architecture Overview

Amodal is a layered system. The SDK provides the core reasoning engine, the Runtime serves it over HTTP, and the CLI provides the developer interface. Each layer has a clear boundary and a clear job. The SDK never knows about HTTP. The Runtime never knows about the CLI. The same reasoning engine adapts to different deployment contexts without code changes.

This separation is deliberate. The SDK can run embedded in a CLI, behind an HTTP server, inside a cloud function, or in a test harness. The same reasoning engine adapts to different deployment contexts without code changes.

## Data Flow: What Happens When a User Sends a Message

When a user types a message and hits enter, here is what happens — from keystroke to streamed response:

### 1. Client to Runtime

The chat client (CLI, web app, or widget) sends a POST to `/chat/stream` with the message text and an optional `sessionId`. If this is a new conversation, no session ID is sent. The connection stays open — the response streams back over SSE.

### 2. Session Manager

The runtime's session manager either creates a new session or loads an existing one. For a new session, it generates a unique ID and initializes an empty message history. For an existing session, it loads the conversation history from the store (PGLite or Postgres). The session manager also checks that the session has not expired.

### 3. Context Compiler

Before the message reaches the LLM, the context compiler assembles the full prompt. This is where the layered config becomes a single, coherent prompt:

- **System prompt**: Built from the agent's role definition, `userContext` from `amodal.json`, and active skill instructions
- **Knowledge index**: A compact listing of all available KB documents (titles, tags, categories) so the agent knows what it can load on demand
- **Tool definitions**: The complete tool list — built-in tools, custom tools, MCP tools — filtered by the caller's role
- **Conversation history**: Previous messages in this session, potentially compacted if the context window is getting full
- **Current message**: The user's new message

The assembled prompt is validated against the model's context window. If it exceeds the limit, the context compiler triggers compaction — summarizing older conversation turns while preserving recent context and any active tool call chains.

### 4. SDK: The ReAct Loop

The SDK sends the assembled prompt to the configured LLM provider and enters the ReAct (Reason + Act) loop:

1. **Reason**: The LLM generates a response. This might be a direct text answer, or it might include one or more tool calls.
2. **Act**: If tool calls are present, the SDK executes them — making API requests, loading knowledge documents, dispatching task agents, rendering widgets. Each tool result is appended to the context.
3. **Loop**: The updated context (with tool results) goes back to the LLM. The model reasons about the results and decides whether to make more tool calls or produce a final response.

This loop continues until the model produces a response with no tool calls (it is done), or until a safety limit is hit (max tool calls per session, timeout).

### 5. Tool Execution

Tool calls execute through the SDK's tool executor. Each tool type has its own execution path:

- **request**: The SDK resolves the target connection's auth config, injects credentials, makes the HTTP call, and returns the response. Write intents trigger a confirmation event back to the client.
- **dispatch**: A new task agent is created with its own fresh context and the explore model. It runs its own mini ReAct loop, then returns a compressed summary to the primary agent.
- **load_knowledge**: The SDK loads the requested KB document from the config snapshot and injects it into the current context.
- **shell_exec**: Runs the command locally (if enabled in the sandbox config).
- **present**: The SDK emits a widget event over SSE with the structured data for the client to render.

### 6. Response Streaming

Throughout this entire process, the runtime streams events to the client over SSE. The client receives `text_delta` events as the LLM generates text, `tool_call_start` and `tool_call_result` events as tools execute, `subagent_event` events when task agents are working, and a final `done` event when the response is complete.

The client renders these events in real time. The user sees the agent thinking, calling tools, and composing its answer — not a loading spinner followed by a wall of text.

## Deployment

### Local Development

```bash
amodal dev    # starts runtime reading from local git repo
amodal chat   # connects to local runtime
```

The runtime reads your config directory directly from the filesystem. Changes to files are hot-reloaded — edit a skill, and the next message uses the updated version. This is the fastest feedback loop for development.

### Production

For production, run the runtime as a standalone server or in a Docker container:

```bash
amodal serve          # run from local config
amodal docker build   # build a Docker image
```

The git repo is the source of truth. Everything about your agent — connections, skills, knowledge, automations, and config — lives in version-controlled files.

## Security Boundaries

### What Runs Where, What Has Access to What

The security model is built around isolation at every layer:

**The SDK** runs inside the runtime process. It has access to the resolved config (including decrypted secrets in memory), the LLM provider APIs, and the connected system APIs. Secrets are decrypted from environment variables at parse time and held only in memory — they are never written to disk, logged, or included in LLM prompts.

**The Runtime** manages sessions and tool execution. The runtime itself does not store secrets — it receives them from the environment and passes them to the SDK.

### How Secrets Flow

Secrets never appear in deployment snapshots, API responses, LLM prompts, or logs.

1. You set secrets as environment variables in your hosting environment (Kubernetes secrets, Fly.io secrets, etc.)
2. `amodal.json` references them with `env:VARIABLE_NAME`
3. The runtime resolves `env:` references at startup, holding the values in memory
4. The SDK uses them for API authentication — injecting headers, signing requests — but never includes them in the context sent to the LLM
5. The scrubbing layer in the SDK scans all outbound context for patterns that look like secrets (API keys, tokens, connection strings) and redacts them. This is a defense-in-depth measure — secrets should not be in the context at all, but if they leak through a tool result, the scrubber catches them.

### Role-Based Tool Filtering

Not every caller should see every tool. The SDK filters the tool list based on the caller's role before the LLM sees it:

- **User role**: Sees all non-admin tools.
- **Admin role**: Sees everything including KB management, connection configuration, and automation control.
- **Automation role**: Sees read-only tools by default. Write tools are only available if `writeEnabled` is true for the specific automation.

This filtering happens at the SDK layer, before the prompt is assembled. The LLM never knows about tools it cannot use — they are simply absent from its tool list.

## Key Protocols

| Protocol | Where | Purpose |
|----------|-------|---------|
| **SSE** (Server-Sent Events) | Runtime to Client | Streaming chat responses, tool calls, widget events |
| **REST** | Runtime API | Session management, automation control |
| **WebSocket** | N/A | Not used — SSE handles all streaming |

SSE was chosen over WebSockets for streaming because it is simpler (unidirectional), works through more proxies and CDNs, and automatically reconnects on connection drops. The client only needs to send messages (POST requests) and receive events (SSE stream) — there is no bidirectional protocol to manage.
